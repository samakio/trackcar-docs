# TrackCar Installer Admin 요구사항 및 기능 명세서

- 작성일: 2026-03-25
- 버전: v1.0
- 상태: 초안

---

## 1. 문서 개요

### 1.1 목적

본 문서는 DTG 설치기사 및 내부 운영자가 사용하는 **Installer Admin Web**의 MVP 요구사항과 기능을 정의한다. 본 시스템은 DTG 설치 이후 실제 서비스 개통까지 필요한 최소 절차인 **Owner 계정 생성, 사용자 초대, DTG 등록, 차량 등록, 기사 등록, 차량/기사/기기 매핑, 연동 검증, 모바일 앱 활성화 확인**을 웹 기반으로 처리하는 것을 목표로 한다.

### 1.2 범위

본 문서의 범위는 다음과 같다.

- 설치 완료 차량 현황을 차트로 조회하는 대시보드
- Owner 계정 관리
- Member/Driver 사용자 계정 연결 관리
- DTG 기기 관리
- 차량 관리
- 기사 관리
- 그룹 관리(최소)
- 매핑 관리
- 연동 검증
- 모바일 앱 초대/최초 로그인 상태 확인
- 최소한의 시스템 관리(사용자/권한/공통코드)

다음 항목은 MVP 범위에서 제외한다.

- 설치 작업 배정/일정/작업 관리
- 재방문/교체/철거 관리
- 사진 첨부 및 전자서명
- 정산/과금 기능
- 고급 통계 및 BI 리포트
- 기사 모바일 단말 다건 등록의 화면 기능

단, **데이터 모델은 향후 복수 기사/복수 모바일 단말 확장을 고려한 구조**를 전제로 한다.

### 1.3 개정 배경

본 문서는 기존 요구사항 정의서와 기능 정의서를 통합한 버전이다. 보완의 핵심은 다음과 같다.

- 모바일 앱과 Installer Admin 간 역할 경계 명확화
- 계정 주체(owner / member / driver) 분리
- 그룹의 마스터 시스템 정의
- 설치 완료와 서비스 활성화 완료 상태 분리
- 앱 초대 및 최초 로그인 플로우 반영
- 향후 복수 기사/교대 운전 시나리오를 고려한 데이터 구조 정리

## 2. 시스템 목표

### 2.1 비즈니스 목표

- DTG 설치 후 서비스 개통 리드타임 단축
- 설치 현장에서 계정/차량/기사 등록 누락 최소화
- 차량-기사-기기 매핑 정확도 향상
- 서버 연동 실패 및 앱 활성화 실패 조기 탐지
- 설치 품질의 표준화
- 모바일 앱 운영 데이터와 설치 데이터의 충돌 최소화

### 2.2 사용자 목표

- 설치기사가 현장에서 빠르게 등록/매핑/검증을 완료할 수 있어야 한다.
- 운영자는 설치 완료 차량 현황과 연동 상태를 대시보드에서 즉시 파악할 수 있어야 한다.
- Owner 및 앱 사용자에게 실제 사용 가능한 계정 상태가 제공되어야 한다.
- 오류 차량 및 미완료 차량을 쉽게 식별할 수 있어야 한다.

### 2.3 통합 목표

- Installer Admin과 모바일 앱은 동일한 마스터 ID 체계를 사용해야 한다.
- 설치 완료와 앱 활성화 완료를 동일 개념으로 취급하지 않아야 한다.
- 그룹, 차량, 기사, 기기 기준정보의 마스터 생성 주체를 명확히 정의해야 한다.

## 3. 시스템 역할 정의

### 3.1 Installer Admin Web의 역할

Installer Admin Web은 다음을 담당한다.

- 고객/Owner 생성
- 기본 그룹 생성 및 관리
- DTG 기기 등록
- 차량 등록
- 기사 등록
- DTG-차량 매핑
- 차량-기사 매핑
- 연동 검증
- 앱 초대/최초 로그인 준비 상태 관리

### 3.2 모바일 앱의 역할

모바일 앱은 다음을 담당한다.

- 운행정보 확인
- 관제/알림 확인
- 사용자 편의 설정 관리
- 사용자 권한 범위 내 그룹/차량/기사 정보 조회
- 운전자코드 입력 및 실시간 운행정보 확인
- 최초 로그인 후 서비스 사용 시작

### 3.3 마스터 시스템 원칙

MVP 기준 마스터 시스템 원칙은 다음과 같다.

- **Owner / Group / Vehicle / Driver / Device의 최초 생성 마스터는 Installer Admin**
- 모바일 앱은 조회 및 제한적 수정만 허용
- DTG 형식승인번호, 제품일련번호, 설치상태, 소유 매핑 정보는 Admin 전용 관리
- 모바일 앱은 사용자 편의 설정, 알림 설정, 사용자 표시 설정 중심으로 동작

## 4. 이해관계자 및 사용자 유형

### 4.1 이해관계자

- 서비스 운영 조직
- 설치 대행/설치기사
- 고객사 또는 차량 소유자(Owner)
- 내부 CS/운영 관리자
- 모바일 앱 실제 사용자(Owner 관리자, 일반 사용자, 기사)

### 4.2 사용자 유형

#### A. Installer

현장에서 실제 설치 및 최초 등록을 수행하는 사용자

주요 업무: Owner 조회/생성, 그룹 생성/확인, DTG 등록, 차량 등록, 기사 등록, 사용자 초대 준비, 즉시 매핑, 연동 검증 실행

#### B. Operator

설치 후 데이터 정합성, 오류 대응, 재검증을 수행하는 사용자

주요 업무: 전체 데이터 조회, 수정 및 보완, 연동 실패 대응, 미매핑 상태 정리, 앱 초대/활성화 상태 점검

#### C. Admin

시스템 운영 권한 보유 사용자

주요 업무: 사용자/권한 관리, 공통코드 관리, 전체 데이터 관리, 감사 및 정정 이력 확인

#### D. Owner User / Member User / Driver User

모바일 앱을 실제 사용하는 사용자

주요 업무: 계정 로그인, 연결 차량 및 DTG 상태 확인, 운행정보 확인, 알림 수신, 필요 시 운전자코드 입력

## 5. 계정 및 권한 모델

### 5.1 계정 모델 분리

시스템은 다음 계정 개념을 구분해야 한다.

#### Owner Account

- 고객사 대표 또는 개인 소유주 계정
- 계약/소유의 최상위 주체
- 그룹 및 사용자 관리의 상위 주체

#### Member Account

- 회사 관리자, 배차 담당자, 현장 담당자 등 일반 운영 사용자
- Owner 소속으로 권한 부여
- 모바일 앱 또는 향후 관리 화면 사용 가능

#### Driver User

- 실제 운전자와 연결되는 사용자
- Driver 엔터티와 앱 로그인 계정을 분리 가능해야 함
- MVP에서는 1:1 연결을 기본으로 하되 구조상 분리 보존

### 5.2 권한 정책

- Installer는 생성/등록/매핑/검증 권한을 가진다.
- Operator는 전체 조회/수정/재검증 권한을 가진다.
- Admin은 시스템 및 공통코드, 권한 정책을 관리한다.
- Owner/Member/Driver 계정은 모바일 앱 사용 권한을 가진다.

### 5.3 앱 초대 정책

- 계정 생성 후 모바일 앱 초대 상태를 관리해야 한다.
- 초대 방식은 SMS 링크 또는 임시 비밀번호 발급 방식을 지원해야 한다.
- 최초 로그인 완료 여부를 별도 상태로 관리해야 한다.
- 약관/개인정보 동의 여부 저장은 차후 상세 정책과 연계 가능해야 한다.

## 6. 핵심 업무 시나리오

### 시나리오 1. 신규 고객 현장 설치

1. 설치기사가 Owner를 검색한다.
2. 기존 계정이 없으면 Owner를 신규 생성한다.
3. 기본 그룹이 없으면 Default Group을 자동 생성한다.
4. DTG 기기를 등록한다.
5. 차량을 등록한다.
6. 기사를 등록한다.
7. 필요 시 모바일 앱 사용자 계정을 생성/초대한다.
8. DTG와 차량을 매핑한다.
9. 차량과 기사를 매핑한다.
10. 연동 검증을 실행한다.
11. 모바일 앱 계정 초대 또는 현장 즉시 로그인 테스트를 진행한다.
12. 설치 완료와 서비스 활성화 완료 상태를 각각 기록한다.

### 시나리오 2. 기존 고객 추가 차량 설치

1. Owner를 기존 데이터에서 조회한다.
2. 기존 그룹을 선택하거나 필요 시 그룹을 추가한다.
3. DTG 기기를 등록한다.
4. 차량을 등록한다.
5. 필요 시 기사 정보를 신규 등록 또는 재사용한다.
6. 필요 시 기존 Member/Driver 계정을 재사용한다.
7. 차량-기사-기기를 매핑한다.
8. 연동 검증 후 설치 완료 처리한다.
9. 앱 활성화가 미완료면 후속 상태로 남긴다.

### 시나리오 3. 연동 실패 확인

1. 운영자가 대시보드에서 Offline 또는 Error 상태 차량을 조회한다.
2. 기기 등록, 매핑, 최근 수신 상태, 앱 활성화 상태를 확인한다.
3. 필요 시 매핑 수정 또는 재검증을 수행한다.
4. 사용자 정보 오류면 정정 이력을 남기고 재검증한다.

### 시나리오 4. 앱 초대 미완료 확인

1. 운영자가 설치 완료 후 앱 활성화 미완료 건을 조회한다.
2. 초대 발송 여부, 최초 로그인 여부, 기본 차량 연결 여부를 확인한다.
3. 필요 시 초대를 재발송한다.

## 7. 상태 모델

### 7.1 설치와 활성화 상태 분리

시스템은 다음 두 개념을 분리해 관리해야 한다.

#### 설치 완료

현장 물리 설치 및 서버 연동 검증 완료 상태

#### 서비스 활성화 완료

모바일 앱 최초 로그인 및 실제 서비스 사용 가능 상태

### 7.2 공통 상태값

공통 온보딩 상태값은 다음을 기준으로 관리한다.

- ACCOUNT_CREATED
- GROUP_READY
- DEVICE_REGISTERED
- VEHICLE_REGISTERED
- DRIVER_REGISTERED
- DEVICE_CONNECTED
- VEHICLE_MAPPED
- DRIVER_MAPPED
- APP_INVITED
- APP_FIRST_LOGIN_DONE
- LIVE

### 7.3 검증 상태값

- NOT_CHECKED
- CHECKING
- PASS
- WARNING
- FAIL

### 7.4 DTG 상태값

| 상태 코드 | 의미 | 비고 |
|---|---|---|
| PENDING_ACTIVATION | 인증서 등록 완료, 서버 검증 대기 중 | JITR 후 초기 상태 |
| REGISTERED | DTG 등록 완료 | 물리 설치 전 상태 |
| ACTIVE | 서버 검증 통과, 데이터 수신 가능 | ingest-auth-validate 통과 후 |
| MAPPED | 차량과 매핑됨 | - |
| ONLINE | 최근 데이터 수신 중 | - |
| OFFLINE | 장기 미수신 | - |
| ERROR | 장치 이상 상태 | - |

## 8. 기능 요구사항

## 8.1 대시보드

### 목적

설치 완료 차량 및 DTG 운영 상태, 앱 활성화 현황을 차트와 지표로 한눈에 파악한다.

### 주요 기능

- 총 설치 차량 수를 표시해야 한다.
- 활성 DTG 수를 표시해야 한다.
- 미연결/오류 DTG 수를 표시해야 한다.
- 차량-기사 매핑 완료 수를 표시해야 한다.
- 연동 검증 완료 수를 표시해야 한다.
- 앱 활성화 완료 수를 표시해야 한다.
- 설치 차량 추이 차트를 제공해야 한다.
- DTG 연동 상태 차트를 제공해야 한다.
- 매핑 상태 차트를 제공해야 한다.
- 앱 초대/활성화 상태 차트를 제공해야 한다.
- 최근 이상 차량 목록을 제공해야 한다.

### 필터링

- 기간, Owner 유형, 그룹, DTG 모델, 검증 상태, 앱 활성화 상태로 필터링할 수 있어야 한다.

---

## 8.2 Owner 계정 관리

### 목적

개인/법인 Owner 계정을 생성 및 조회한다.

### 주요 기능

- 개인/법인 유형을 구분할 수 있어야 한다.
- 이름 또는 회사명을 등록할 수 있어야 한다.
- 휴대폰번호를 등록할 수 있어야 한다.
- 사업자번호를 등록할 수 있어야 한다.
- 기존 휴대폰번호 또는 사업자번호 중복 여부를 확인할 수 있어야 한다.
- Owner별 등록 차량 수와 기사 수를 조회할 수 있어야 한다.
- Owner별 기본 그룹 생성 상태를 조회할 수 있어야 한다.
- Owner 하위 사용자(Member/Driver) 수를 조회할 수 있어야 한다.
- 법인 Owner는 **운송사업자등록번호**를 등록할 수 있어야 한다. (eTAS 전송용)

### 입력 항목

- Owner 유형(개인/법인)
- 이름 또는 회사명
- 휴대폰번호
- 이메일(optional)
- 사업자번호(optional)
- 운송사업자등록번호(optional, 법인Owner 필수) — eTAS 전송 시 필요

### 참고

운송사업자등록번호는 한국교통안전공단 eTAS 전송 시 차량별 1일 운행기록의 사업자 메타 보강에 사용된다. ([별표 2], [별표 3] 참조)

---

## 8.3 그룹 관리

### 목적

모바일 앱과 공통 사용하는 기준 그룹을 생성 및 관리한다.

### 주요 기능

- Owner 생성 시 Default Group을 자동 생성할 수 있어야 한다.
- 그룹은 Installer Admin에서 생성/수정할 수 있어야 한다.
- 차량, 기사, 사용자 계정은 그룹에 소속될 수 있어야 한다.
- 모바일 앱은 해당 그룹 정보를 참조할 수 있어야 한다.

---

## 8.4 Member 관리

### 목적

## 8.6 DTG 기기 관리

### 목적

설치된 DTG의 식별정보와 연결 상태를 관리한다.

### 주요 기능

- 시리얼번호를 등록할 수 있어야 한다.
- 형식승인번호를 등록할 수 있어야 한다.
- 제품일련번호를 등록할 수 있어야 한다.
- 모델명과 펌웨어 버전을 등록할 수 있어야 한다.
- 통신 회선 정보를 등록할 수 있어야 한다.
- DTG 상태를 REGISTERED / MAPPED / ONLINE / OFFLINE / ERROR로 관리할 수 있어야 한다.
- 최근 수신시각을 조회할 수 있어야 한다.
- 특정 DTG가 현재 어느 차량과 매핑되었는지 조회할 수 있어야 한다.
- DTG 관리 항목 중 형식승인, 제품일련번호, 설치상태는 Admin 전용 수정 대상으로 구분할 수 있어야 한다.

### 입력 항목

- 시리얼번호
- 형식승인번호
- 제품일련번호
- 모델명
- 펌웨어 버전(optional)
- 통신 회선 번호(optional)
- 설치 메모(optional)

---

## 8.7 차량 관리

### 목적

서비스 대상 차량을 등록 및 관리한다.

### 주요 기능

- 차량번호, VIN, 차종, 소속 Owner, 소속 Group을 등록할 수 있어야 한다.
- 차량의 현재 연결 DTG, 기본 기사, 앱 활성화 상태를 조회할 수 있어야 한다.
- 향후 복수 기사 연결 확장을 고려한 구조여야 한다.
- **자동차유형코드**를 등록할 수 있어야 한다. (eTAS 전송용)
- 차량번호는 **자동차등록번호표기 규격**([별표 4])에 맞게 검증해야 한다.

### 입력 항목

- 차량번호 (형식: `[별표 4]` 기준, 최대 20자)
- 차대번호(VIN) (17자리, 영문+숫자)
- **자동차유형코드** (eTAS 전송용, 코드셋 관리 필요)
- 차종
- 연식(optional)
- owner
- group

### 참고

- VIN은 17자리 alphanumeric 문자열로, 중복검증 및 형식검증(문자셋)을 수행해야 한다.
- 자동차유형코드는 eTAS 전송 시 `[별표 2]` 운행기록 배열순서의 자동차유형 필드에 매핑된다.
- 차량번호 형식 검증은 `[별표 4]` 자동차등록번호의 표기 규격을 따른다.

---

## 8.8 기사 관리

### 목적

실제 운전자를 등록 및 관리한다.

### 주요 기능

- 기사명, 휴대폰번호, 기사코드/사번, 소속 Owner, 소속 Group을 등록할 수 있어야 한다.
- 기사 엔터티와 앱 로그인 계정을 연결할 수 있어야 한다.
- MVP에서는 기본 기사 1명 연결을 지원하되, 향후 복수 기사 연결 확장 가능 구조여야 한다.
- **운전자코드**를 등록할 수 있어야 한다. (eTAS 전송용, `[별표 2]` 배열순서)

### 운전자코드 형식

- 형식: alphanumeric, 최대 20자 (구체적 자릿수는 `[별표 2]` 기준)
- 중복 검증 및 형식 검증을 수행해야 한다.
- 운전자코드는 eTAS 전송 시 Trip 데이터의 운전자 식별에 사용된다.

### 참고

DTG 표준 상태 필드는 `ignition`(시동 상태)을 단일 기준으로 사용한다. (`engineOn` 속성은 사용하지 않음)

## 8.9 매핑 관리

### 목적

차량/기사/DTG의 설치 관계를 완성한다.

### 하위 기능

#### A. DTG-차량 매핑

- DTG 선택
- 차량 선택
- 설치일시 저장
- 활성 여부 설정

#### B. 차량-기사 매핑

- 차량 선택
- 기사 선택
- 기본 운전자 여부 설정
- 적용 시작일시 저장

#### C. Quick Mapping

- 차량 검색 또는 신규 등록
- 기사 검색 또는 신규 등록
- DTG 검색
- 즉시 저장
- 연동 검증 바로 실행

### 요구사항

- DTG는 동시에 1대 차량에만 활성 연결 가능해야 한다.
- 차량도 동시에 1개 활성 DTG만 허용해야 한다.
- 차량에는 기본 기사 1명을 지정할 수 있어야 한다.
- 구조적으로는 복수 기사 매핑 이력을 저장할 수 있어야 한다.

- 설치 완료와 서비스 활성화 완료 상태를 구분해 저장한다.

## 8.12 연동 검증

### 목적

설치 이후 서버 연동 및 앱 사용 준비 상태를 검증한다.

### 검증 항목

- 기기 등록 여부를 확인할 수 있어야 한다.
- DTG-차량 매핑 여부를 확인할 수 있어야 한다.
- 차량-기사 매핑 여부를 확인할 수 있어야 한다.
- 최근 N분 이내 데이터 수신 여부를 확인할 수 있어야 한다.
- GPS 수신 여부를 확인할 수 있어야 한다.
- 앱 초대 상태 및 최초 로그인 여부를 확인할 수 있어야 한다.

### 판정

- 검증 결과를 PASS / WARNING / FAIL로 판정할 수 있어야 한다.
- 재검증 기능을 제공해야 한다.
- 정정 후 재검증 이력을 남겨야 한다.
