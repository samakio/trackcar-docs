# Installer Admin Web 통합테스트 시나리오

- 작성일: 2026-03-31
- 상태: Draft
- 대상 시스템: `installer-admin-web`, `aws-functions`, `dtg-device-simulator`, AWS IoT Core, ingest 파이프라인

---

## 1. 문서 목적

본 문서는 `dtg-device-simulator`를 실제 DTG 장비처럼 사용하여, TrackCar의 설치 운영 흐름이 정상 동작하는지 검증하기 위한 통합테스트 시나리오를 정의한다.

검증 범위:
- DTG 시뮬레이터의 실제 MQTT 발행
- AWS IoT Core 수신
- ingest 파이프라인 처리
- Installer Admin Web의 고객사/장치/차량/기사/매핑/연동 검증
- 법인 고객사와 개인 고객사 흐름 차이

---

## 2. 공통 사전조건

### 2.1 환경 조건

- `installer-admin-web`는 실제 API Gateway를 바라보는 환경이어야 한다.
- 실제 Cognito 로그인 세션으로 접속해야 한다.
- `aws-functions` / `aws-infra` / `installer-admin-web` 최신 배포 상태여야 한다.
- AWS IoT Core rule과 ingest Lambda가 정상 동작해야 한다.

### 2.2 DTG 시뮬레이터 조건

- `dtg-device-simulator`는 `iot` 모드로 실행한다.
- MQTT topic은 `dtg/{deviceId}/telemetry`를 사용해야 한다.
- 시뮬레이터가 사용하는 `deviceId`와 관리자 화면에서 매핑하는 장치의 내부 `device_id` 계약이 맞아야 한다.
- 최소 1회 이상 batch flush가 일어날 수 있도록 충분한 시간(기본 30초 이상)을 확보한다.

### 2.3 중요한 주의사항

고객사명, 차량번호, 장치 시리얼번호, 기사명 등은 본 문서의 **예정된 테스트 값**을 사용한다.

단, DTG 시뮬레이터의 실제 `deviceId`와 서버 매핑 기준 `device_id`는 현재 구현상 별도 정렬 확인이 필요하다.
이 계약이 맞지 않으면 연동 검증은 실패할 수 있다.

## 3. 공통 합격 기준

### 3.1 UI 기준

- 화면에서 등록/매핑/검증 흐름이 막힘 없이 이어져야 한다.
- 에러가 발생하면 원인 파악 가능한 메시지가 보여야 한다.
- 검증 결과는 최소 아래 항목을 표시해야 한다.
  - 장치 등록
  - 차량 매핑
  - 기사 배정
  - GPS 수신
  - 하트비트 수신
  - 마지막 수신 시각

### 3.2 데이터 기준

- 등록한 고객사, 차량, DTG, 기사가 실제 DB/API에서 조회 가능해야 한다.
- 매핑 이후 검증 대상 차량 검색이 가능해야 한다.
- DTG 수신이 실제로 들어오면 검증 결과에 반영되어야 한다.

### 3.3 판정 기준

- `PASS`: 최근 데이터 수신 정상
- `WARNING`: 일부 연결은 됐지만 상태 미완전
- `FAIL`: 수신 없음 또는 핵심 연결 미완료

> 주의: 최종 `PASS`만 보지 말고 `driver_mapped`, `gps_check`, `heartbeat_check` 등 개별 항목도 함께 확인한다.

## 4. 시나리오 1 — 법인 고객사 설치 온보딩 E2E

### 4.1 목적

법인 고객사를 기준으로 Installer Admin Web의 표준 설치 흐름이 끝까지 정상 동작하는지 검증한다.

### 4.2 테스트 데이터

#### 고객사
- 고객사 유형: `BUSINESS`
- 고객사명: `통합테스트운수A`
- 사업자등록번호: `221-81-90001`
- 운송사업자등록번호: `1234567890`
- 휴대폰번호: `010-9000-0001`
- 이메일: `owner.biz.a@trackcar-test.co.kr`
- 기본 그룹명: `통합테스트운수A 기본그룹`

#### 차량
- 차량번호: `경기45바6789`
- VIN: `KMHAT41BPXA000001`
- 차량유형: `TRUCK`

#### 기사
- 기사명: `홍길동A`
- 휴대폰번호: `010-9100-0001`
- 운전자코드: `DRV-BIZ-A-001`

#### DTG 장치
- 장치 시리얼번호(표시용 기준값): `DTG-BIZ-A-001`
- 형식승인번호: `APPROVAL-BIZ-A-001`
- 제품일련번호: `PROD-BIZ-A-001`

### 4.3 필수 사전 정렬 단계

#### DTG identity 정렬

현재 구현상 시뮬레이터의 `deviceId`는 랜덤 생성될 수 있으므로, 테스트 실행 전 아래를 먼저 확인한다.

1. 시뮬레이터 차량 1대 생성
2. 반환된 `deviceId` 확인
3. 서버가 바인딩 조회에 쓰는 `device_id`와 실제로 일치하는지 확인

**사전 체크포인트**
- 시뮬레이터가 발행하는 `deviceId`
- 서버가 바인딩 조회에 쓰는 `device_id`
- 이 둘이 같은지 확인

### 4.4 테스트 절차

#### Step 1. DTG 시뮬레이터 실행
- `dtg-device-simulator`를 `iot` 프로필로 실행
- 차량 1대 생성
- 반환값 기록
  - `deviceId`
  - `vehicleId`
  - `plateNumber`

**기대 결과**
- 시뮬레이터가 AWS IoT Core로 publish 준비 완료 상태여야 한다.
- 연결 실패 로그가 없어야 한다.

#### Step 2. 법인 고객사 생성
입력값:
- `통합테스트운수A`
- `221-81-90001`
- `1234567890`
- `010-9000-0001`
- `owner.biz.a@trackcar-test.co.kr`

**기대 결과**
- 고객사 생성 성공
- 기본 그룹 자동 생성
- OWNER 앱 계정 자동 생성
- 기사 자동 생성 없음

**검증 포인트**
- 고객사 상세에서 OWNER 앱 계정 존재 확인
- 그룹 목록에서 기본 그룹 존재 확인

#### Step 3. DTG 장치 등록
입력값:
- `DTG-BIZ-A-001`
- `APPROVAL-BIZ-A-001`
- `PROD-BIZ-A-001`

**기대 결과**
- 장치 등록 성공

#### Step 4. 차량 등록
입력값:
- `경기45바6789`
- `KMHAT41BPXA000001`
- 그룹: `통합테스트운수A 기본그룹`

**기대 결과**
- 차량 등록 성공

#### Step 5. 기사 등록
입력값:
- `홍길동A`
- `010-9100-0001`
- `DRV-BIZ-A-001`

**기대 결과**
- 기사 등록 성공

#### Step 6. 매핑 저장
- DTG 장치
- 차량
- 기사
를 Quick Mapping으로 연결

**기대 결과**
- DTG-차량 매핑 성공
- 차량-기사 매핑 성공

#### Step 7. 연동 검증 대상 검색
- 연동 검증 메뉴 진입
- `경기45바6789`
- `DTG-BIZ-A-001`
- `홍길동A`
로 각각 검색

**기대 결과**
- 방금 매핑한 차량이 자동완성 후보에 노출
- 후보는 ACTIVE 장치 바인딩 차량만 보여야 함

#### Step 8. 선택 차량 검증 실행
- 방금 매핑한 차량 1대를 선택해 검증 실행

**기대 결과**
- 검증 요청 성공
- 검증 목록에 해당 차량 결과 생성
- 상세 패널 조회 가능

#### Step 9. 검증 결과 확인
확인 항목:
- 장치 등록
- 차량 매핑
- 기사 배정
- GPS 수신
- 하트비트 수신
- 마지막 수신 시각

**합격 기준**
- 시뮬레이터가 실제로 데이터 발행 중이면 `last_received_at` 표시
- 최근 수신이면 `PASS`

## 5. 시나리오 2 — 개인 고객사 설치 온보딩 E2E

### 5.1 목적

개인 고객사 생성 시 자동 생성 규칙과 최소 매핑/검증 흐름이 정상 동작하는지 검증한다.

### 5.2 테스트 데이터

#### 고객사
- 고객사 유형: `PERSONAL`
- 고객사명: `권동원P`
- 사업자등록번호: `331-12-90002`
- 운송사업자등록번호: `2345678901`
- 휴대폰번호: `010-9000-0002`
- 이메일: `owner.personal.p@trackcar-test.co.kr`
- 기본 그룹명: `권동원P 기본그룹`

#### 차량
- 차량번호: `서울78다9012`
- VIN: `KMHAT41BPXA000002`
- 차량유형: `TRUCK`

#### 자동 생성 기사 기대값
- 기사명: `권동원P`
- 기사 전화번호: `010-9000-0002`
- 운전자코드 패턴: `DRV-{ownerId앞8자리}`

#### DTG 장치
- 장치 시리얼번호(표시용 기준값): `DTG-PER-A-001`
- 형식승인번호: `APPROVAL-PER-A-001`
- 제품일련번호: `PROD-PER-A-001`

### 5.3 핵심 검증 포인트

개인 고객사는 생성 직후 아래 3개가 자동 생성되어야 한다.
- 기본 그룹
- OWNER 앱 계정
- 기사

즉 이 시나리오의 핵심은 **자동 생성 규칙 검증**이다.

### 5.4 테스트 절차

#### Step 1. 개인 고객사 생성
입력값:
- `권동원P`
- `331-12-90002`
- `2345678901`
- `010-9000-0002`
- `owner.personal.p@trackcar-test.co.kr`

**기대 결과**
- 고객사 생성 성공
- 기본 그룹 자동 생성
- OWNER 앱 계정 자동 생성
- 기사 자동 생성

#### Step 2. 자동 생성 결과 확인
확인 대상:
- 그룹 목록
- 고객사 상세
- 기사 관리

**기대 결과**
- 그룹명 = `권동원P 기본그룹`
- OWNER 계정 존재
- 자동 생성 기사 존재
- 기사명/전화번호가 고객사 입력값과 맞음

#### Step 3. 차량 등록
입력값:
- `서울78다9012`
- `KMHAT41BPXA000002`

#### Step 4. DTG 장치 등록
입력값:
- `DTG-PER-A-001`
- `APPROVAL-PER-A-001`
- `PROD-PER-A-001`

#### Step 5. 매핑
- 자동 생성 기사 사용
- 기본 그룹 사용
- DTG-차량-기사 매핑

#### Step 6. 연동 검증
- `서울78다9012`
- `DTG-PER-A-001`
- `권동원P`
로 검색 후 검증 실행

**합격 기준**
- 자동 생성 기사 기준으로도 검증 성공
- `driver_mapped` 정상
- telemetry 수신 시 `PASS`

## 6. 공통 실패/음수 체크

### 6.1 검증 후보 노출 조건
아래는 후보에 나오면 안 된다.
- 매핑 안 된 차량
- ACTIVE 장치 바인딩 없는 차량

### 6.2 선택 검증 조건
- 차량 선택 없이 실행하면 안 된다.

### 6.3 데이터 계약 확인
- 시뮬레이터 발행 `deviceId`
- 서버 binding 조회 기준 `device_id`
- 이 둘이 안 맞으면 검증 실패

## 9. 추가 권장 시나리오

### 시나리오 3 — 기존 고객사에 차량 1대 추가 설치

실제 운영 빈도를 고려하면 아래 시나리오도 중요하다.
- 기존 고객사 재사용
- 기존 그룹 재사용
- 장치/차량 추가 등록
- 매핑/검증 수행

이 시나리오는 신규 생성 happy path보다 **중복 방지 / 기존 데이터 재사용 / 운영 보정 흐름**을 더 잘 검증한다.
